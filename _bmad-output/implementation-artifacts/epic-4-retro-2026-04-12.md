## Challenges

### 1. The Happy-Path Gap: Tests Pass, Real Client Does Not

**What happened:** All 196 ACs are covered. Godog E2E passes. matrix-js-sdk smoke test passes. k6 passes. Yet when Phil tested the actual user journey — login → join room → send message → receive message — with a real Matrix client (FluffyChat), it did not work.

**Root cause:** There is a structural gap between protocol-level test coverage and real client compatibility. matrix-js-sdk is a library, not a client. FluffyChat is built on assumptions that library-level HTTP tests do not exercise:
- `/sync` long-polling response format details (missing/malformed fields cause silent client failure)
- Event timeline reconstruction (ordering, `prev_batch`, `limited` flag)
- Auth discovery path (SSO discovery endpoint format differs by client)
- `origin_server_ts` field type (Integer vs. String — clients fail silently on type mismatch)

**Phil's principle:** *Automated tests validate protocol compliance. Whether the goal is achieved — "users can chat" — is only validated by a human using a real client.*

**This is the most important lesson of Epic 4.**

### 2. Code Review MAJORs Still Found in Early Stories

Stories 4-2, 4-3, 4-4 each had MAJOR code review findings:
- 4-2: `ON CONFLICT DO NOTHING` prevented rejoin after leave
- 4-3: `Map.new/1` lost key ordering for maps >32 keys (ATDD miss — no failing test existed for this edge case)
- 4-4: Signing payload incorrectly included `event_id`; ETS table creation lacked restart guard

TEA reduced the frequency of late-discovered architectural issues, but some domain-specific edge cases (canonical JSON behavior at >32 keys) are not naturally covered by AC-driven tests. The acceptance criteria must explicitly describe edge cases to produce tests that catch them.

### 3. ETS Dedup Growth — Unaddressed Technical Debt

`:NebuTxnDedup` grows unbounded. No TTL pruning strategy was implemented. Under high load (k6 500 VUs) this did not surface — but it is a time-bomb for long-running deployments. Deferred to Epic 5 backlog.

---

## Key Insights

1. **Happy-path testing with a real client is a quality gate, not an optional extra.** If the goal is "users can chat via a standard Matrix client," then a human running FluffyChat against the stack is the only true validation of that goal. This must be a defined acceptance criterion for every endpoint epic going forward.

2. **TEA raises the floor, not the ceiling.** The traceability gate prevents untested ACs from shipping. It does not automatically produce tests for edge cases not expressed in ACs. AC quality determines test quality.

3. **ETS application-ownership pattern is now stable.** Crash/restart tests passed in 4-2, 4-4, 4-5, 4-7, 4-13. The pattern is well-understood and reusable in Epic 5+.

4. **The multi-layer test pyramid is complete for the Matrix API layer.** Epic 5 (Compliance) operates mostly at the API layer — the existing test infrastructure (Godog, httptest, ExUnit) is directly reusable.

## Patterns Established (for use in Epic 5+)

| Pattern | Location | Description |
|---|---|---|
| ETS table owned by Application | `core/apps/*/lib/*/application.ex` | Table created in `Application.start/2`, owned by Application process — survives GenServer crashes |
| Room Registry injection | `gateway/internal/matrix/*_handler.go` | `roomRegistryModule()` via `Application.get_env` — enables test isolation without Horde dependency |
| Nebu.EventId.generate/1 | `core/apps/signature/lib/nebu/event_id.ex` | Canonical JSON → SHA-256 → Base64url → `$` prefix — sole permitted event ID source |
| String-keyed DB maps | `core/apps/room_manager/lib/` | All DB-sourced maps use string keys — no atom keys from external data |
| gRPC → Matrix error mapping | `gateway/internal/matrix/*_handler.go` | `status.FromError(err)` → `switch st.Code()` → Matrix JSON error |
| AES-256-GCM media | `media/internal/crypto/` | Upload: encrypt then store; Download: fetch then decrypt; nonce prepended to ciphertext |
