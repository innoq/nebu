The implementation is sound: tokens are generated from `crypto/rand` (32 bytes, base64url), comparison uses `subtle.ConstantTimeCompare`, and the GET-to-POST migration of the logout action eliminates the classic `<img src="/admin/logout">` CSRF vector. ## Findings

### [MEDIUM] `Secure=false` on `csrf_token` cookie

- **CWE / OWASP:** CWE-614 / A05:2021 Security Misconfiguration
- **File:** `gateway/internal/admin/middleware.go:238`
- **Description:** The `csrf_token` cookie is set with `Secure: false`. Over plain HTTP, the cookie can be observed or injected by a network-level attacker (MitM). The code comment references Story 5.15 (HTTPS setup) as the intended fix.
- **Impact:** On a non-TLS deployment, a MitM attacker could read or overwrite the CSRF cookie. Combined with the `SameSite=Strict` attribute and the fact that the Admin UI is typically accessed on a private network, practical exploitation is unlikely but the gap is real.
- **Recommendation:** Track this as a hard dependency on Story 5.15. When HTTPS lands, flip `Secure: true` on both `csrf_token` and `admin_session` cookies simultaneously.
- **Reference:** OWASP ASVS V3.4.1

### [INFO] Legacy callback route `GET /admin/auth/callback` lacks CSRF rotation

- **File:** `gateway/cmd/gateway/main.go:211`
- **Description:** The legacy route `GET /admin/auth/callback` (line 211) is registered without `CSRFMiddleware`, so it does not rotate the CSRF token on OIDC callback completion. The canonical route `GET /admin/callback` (line 220) correctly applies CSRF rotation. The legacy route exists for backward compatibility and will be superseded by Story 3.10.
- **Impact:** If a client uses the legacy callback URL, the CSRF token is not rotated post-login. The existing pre-login token remains valid. This is a defense-in-depth gap, not an exploitable vulnerability, because the pre-login token is already a valid CSRF token.
- **Recommendation:** When Story 3.10 removes legacy routes, this gap disappears. No immediate action required.

### [INFO] Comprehensive CSRF test coverage added

- **Description:** The test suite covers: POST rejection without token, POST rejection with mismatched token, POST acceptance with matching token, GET passthrough, GET issues cookie, token rotation on `/admin/callback`, and context injection.
