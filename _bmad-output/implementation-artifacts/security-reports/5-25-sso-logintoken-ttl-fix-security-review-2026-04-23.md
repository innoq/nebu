## Executive Summary

This change corrects a silent TTL drift: `globalLoginTokens.save()` was called with a hardcoded `5*time.Minute` instead of the documented `30*time.Second`. The fix introduces a named constant `loginTokenTTL = 30 * time.Second`, wires it into the single call site, and removes a stale "MAJOR-2 fix" comment prefix. Four well-structured unit tests (white-box, no sleeps, deterministic via injected timestamps) guard the constant value and the expiry/redemption paths. No new attack surface is introduced. The `ssoStateStore` TTL (10 minutes, for the PKCE authorization code flow) remains correctly independent.

## Findings

### [INFO-1] loginToken replay window reduced from 5 minutes to 30 seconds

- **CWE / OWASP:** CWE-613 (Insufficient Session Expiration) / A07:2021 (Identification and Authentication Failures)
- **File:** `gateway/internal/matrix/sso.go:85` (new constant), `gateway/internal/matrix/sso.go:369` (call site)
- **Description:** The previous `5*time.Minute` TTL gave an attacker a 300-second window to replay a loginToken extracted from browser history, Referer headers, or proxy logs. The new 30-second TTL reduces this to one-tenth of the previous window. Combined with the existing single-use `pop()` semantics (token is deleted on first read), the effective replay window is `min(30s, time-to-first-use)` — typically under 2 seconds in practice.
- **Impact:** Positive. Reduces the theoretical replay window for a URL-borne bearer token from 5 minutes to 30 seconds.
- **Reference:** OWASP ASVS V3.3.1 (session timeout), Matrix Client-Server spec r0.6.1 `m.login.token`.

### [INFO-2] ssoStateStore and loginTokenStore TTLs correctly separated

- **File:** `gateway/internal/matrix/sso.go:282` (`globalSSOState.save(..., 10*time.Minute)`), `gateway/internal/matrix/sso.go:369` (`globalLoginTokens.save(..., loginTokenTTL)`)
- **Description:** The two stores serve distinct purposes: `ssoStateStore` holds PKCE verifiers during the user's interaction with the IdP (requires minutes for password entry, MFA, consent screen), while `loginTokenStore` holds the opaque redirect token for the sub-second client-to-gateway hop. Their TTLs (10 min vs 30 s) are correct for their respective threat models. No cross-contamination — each store has its own type, its own global variable, and its own `pop()` implementation.
- **Impact:** N/A — observation confirming correct design.

### [INFO-3] Test coverage for TTL constant and expiry paths

- **File:** `gateway/internal/matrix/login_token_test.go` (176 lines, 4 tests)
- **Description:** The test suite uses white-box injection (same package) to seed `loginTokenStore.store` with crafted `exp` timestamps, avoiding non-deterministic `time.Sleep`. Tests cover: (1) compile-time constant value assertion, (2) expired-token rejection, (3) valid-token acceptance + single-use removal, (4) integration test that `save()` uses `loginTokenTTL` and produces the correct `exp` range. This is a regression guard — any future change to the TTL value or the `save()` call site will break at least two tests.
- **Impact:** Positive — defense in depth via tests.
