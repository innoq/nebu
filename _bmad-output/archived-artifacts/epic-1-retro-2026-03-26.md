## Challenges

### 1. Versions not defined in planning phase

**What happened:** Go and Elixir versions were not explicitly specified in the Architecture document. `go.mod` initially targeted `go 1.23` while the Dockerfile used `golang:1.26-alpine`. Alpine runtime version (3.19) was incompatible with the Elixir builder (Alpine 3.23 / OpenSSL 3.4), causing a runtime crash in Story 1-9.

**Root cause:** The planning phase defined *which* technologies to use, but not *which versions*. This created downstream friction in Stories 1-1, 1-6, 1-9, and 1-15.

**Impact:** Ad-hoc fixes required in multiple stories. OTP crypto compatibility for Epic 2 (Ed25519/X25519) is also affected — not explicitly documented.

### 2. External tool surprises

**What happened:** Four unexpected tool-level issues required debugging time:
- `buf.build/elixir-protobuf/elixir` remote plugin does not exist on the buf.build registry → fallback to protoc + protoc-gen-elixir (Story 1-6)
- `alpine:3.19` runtime incompatible with `elixir:1.19-alpine` builder due to OpenSSL version mismatch → fix: update runtime to `alpine:3.23` (Story 1-9)
- Elixir umbrella `mix.exs` missing `releases:` config → runtime crash (Story 1-9)
- Go race detector requires CGO / C compiler not present in `golang:1.26-alpine` → `apk add gcc musl-dev` workaround (Story 1-15)

**Root cause:** External tool ecosystem not fully explored during planning. Alpine base image pinning not coordinated with actual builder versions.

### 3. Build container overhead (technical debt, now resolved)

**What happened:** `make proto`, `make test-unit-elixir`, and `make build-core` each install hex, rebar, protobuf, and protoc-gen-elixir on every run (~30–60s overhead). Identified as a LOW action item in Story 1-6 review.

**Resolution during retrospective:** Custom CI Docker images created (`docker/Dockerfile.ci.go`, `docker/Dockerfile.ci.elixir`) with dependencies pre-installed. `.gitlab-ci.yml` updated to use custom images for unit test jobs. See action items below.

### 4. Integration tests included in unit test runs (resolved)

**What happened:** `go test ./...` included `gateway/test/integration/` because no build tag was set. In the CI `unit-tests-go` job (no stack running), the integration tests failed with `connection refused` on `localhost:8080`.

**Secondary issue:** DinD networking — `--add-host=host.docker.internal:host-gateway` does not work in GitLab CI DinD context; the test runner container could not reach compose services.

**Resolution during retrospective:**
- Added `//go:build integration` to both integration test files
- Added `name: nebu` to `docker-compose.yml` (stable network name `nebu_default`)
- Updated `make test-integration` to use `--network=nebu_default` and service names (`gateway:8080`, `core:4000`)
- Added `apk add make` to CI `integration-test` job

---

## Key Insights

1. **Planning phase needs an explicit versions table.** Architecture should include: Go version, Elixir version, OTP version, Alpine version, PostgreSQL version. This prevents version-mismatch friction across all implementation stories.

2. **Small stories stay the right choice for Epic 2.** Epic 2 introduces cryptographic operations (Ed25519/X25519), OIDC JWT validation, and Elixir session management — all domains where catching errors early at story boundaries matters more than reducing story overhead.

3. **CI infrastructure is real code.** CI configuration bugs (wrong network, missing build tags, missing make) should be treated with the same care as application code. Testing the CI pipeline itself is a valid activity.
