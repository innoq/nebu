# Story 1.17: make gen-api — oapi-codegen Build Target

Status: done

## Story

As a developer,
I want `make gen-api` to generate Go types and server interface from the OpenAPI spec,
so that the Admin API always stays in sync with `gateway/api/openapi.yaml` as its single source of truth.

## Acceptance Criteria

1. **Given** `gateway/api/openapi.yaml` exists with a minimal valid spec (one endpoint: `GET /api/v1/health` returning `{"status": "string"}`), **When** `make gen-api` runs, **Then** it executes oapi-codegen inside the Go container and exits code 0.

2. **Given** `make gen-api` completes, **When** `gateway/internal/admin/api_gen.go` is inspected, **Then** it contains a `ServerInterface` with at least one method signature and Go structs for request/response types.

3. **Given** `api_gen.go` is generated, **When** `go build ./...` runs inside the container from `gateway/`, **Then** it compiles successfully.

4. **Given** `gateway/api/openapi.yaml` with an intentionally invalid schema, **When** `make gen-api` runs, **Then** it exits non-zero with a descriptive error message.

5. **Given** the generated file header, **When** `api_gen.go` is opened, **Then** it contains the comment `// Code generated by oapi-codegen — DO NOT EDIT`.

### Makefile Change

**Current (no-op placeholder):**
```makefile
gen-api:
	$(DOCKER_GO) sh -c "cd gateway && go generate ./..."
```

**Target implementation:**
```makefile
gen-api:
	$(DOCKER_GO) sh -c "go run github.com/oapi-codegen/oapi-codegen/v2/cmd/oapi-codegen@latest \
		-generate types,server \
		-package admin \
		-o gateway/internal/admin/api_gen.go \
		gateway/api/openapi.yaml"
```

**Why `@latest` on `go run`?** The `DOCKER_GO` container runs from `/workspace` (project root). There's no `go.mod` at `/workspace` — only `gateway/go.mod`. Using `@latest` makes Go download and run the tool in a temporary module context without touching `gateway/go.mod`. This matches the architecture spec. [Source: architecture.md#API-Spec-First-Workflow]

**Why run from project root (not `cd gateway && go run ...`)?** Running from `/workspace` lets the tool read `gateway/api/openapi.yaml` and write `gateway/internal/admin/api_gen.go` using paths relative to workspace root, consistent with how all other `DOCKER_GO` commands work in this Makefile.

### `gateway/api/openapi.yaml` — Minimal Spec

Create this file (`gateway/api/openapi.yaml`):

```yaml
openapi: "3.0.3"
info:
  title: Nebu Admin API
  version: "0.1.0"
paths:
  /api/v1/health:
    get:
      operationId: GetApiV1Health
      summary: Admin health check
      responses:
        "200":
          description: Health status
          content:
            application/json:
              schema:
                $ref: "#/components/schemas/HealthResponse"
components:
  schemas:
    HealthResponse:
      type: object
      required:
        - status
      properties:
        status:
          type: string
```

**Why this specific spec?** The AC requires one endpoint: `GET /api/v1/health` returning `{"status": "string"}`. Using `operationId: GetApiV1Health` ensures the generated `ServerInterface` method is named `GetApiV1Health`. This aligns with future stories (Epic 6) that will add real Admin API endpoints.

**Remove `.gitkeep`:** After creating `openapi.yaml`, delete `gateway/api/.gitkeep`.

### Expected Generated Output (`api_gen.go`)

The generated file will be placed at `gateway/internal/admin/api_gen.go` in `package admin`. Expected structure:

```go
// Code generated by oapi-codegen — DO NOT EDIT.

package admin

import "net/http"

// HealthResponse defines model for HealthResponse.
type HealthResponse struct {
    Status string `json:"status"`
}

// ServerInterface must be implemented by handlers.
type ServerInterface interface {
    // GetApiV1Health (GET /api/v1/health)
    GetApiV1Health(w http.ResponseWriter, r *http.Request)
}
```

**Package collision check:** `gateway/internal/admin/metrics.go` is already `package admin`. The generated `api_gen.go` will be in the same package. No name conflicts expected — `metrics.go` defines `Metrics`, `NewMetrics`, `coreStateProvider`, `statusRecorder`. Generated names will be `HealthResponse`, `ServerInterface`, `GetApiV1Health`. No overlap.

### Handling Runtime Deps After Generation

Depending on the version of oapi-codegen, the generated code may import `github.com/oapi-codegen/runtime`. If it does, `go build ./...` will fail with a missing module error.

**Fix:** Run `go mod tidy` inside the Go container after generation:
```bash
docker run --rm -v $(PWD):/workspace -w /workspace/gateway golang:1.26-alpine go mod tidy
```

Commit the updated `gateway/go.mod` and `gateway/go.sum` alongside the generated `api_gen.go`.

**How to detect:** After `make gen-api`, check `api_gen.go` imports. If `github.com/oapi-codegen/runtime` is present, run `go mod tidy`.

### `api_gen.go` Should Be Committed

Unlike some codegen workflows where generated files are gitignored, `api_gen.go` **must be committed** to the repository. Rationale: this is how the rest of the codebase refers to `ServerInterface` without requiring every dev to run codegen. Future stories will implement `ServerInterface` from this file.

**Do NOT add `gateway/internal/admin/api_gen.go` to `.gitignore`.**

### Build Verification

After generating `api_gen.go`, verify compilation with:
```bash
$(DOCKER_GO) sh -c "cd gateway && go build ./..."
```

This must exit 0. The currently building gateway in `cmd/gateway/main.go` imports `github.com/nebu/nebu/internal/admin` (for `admin.NewMetrics`). The presence of `api_gen.go` adds new exported types to that package — it must not break existing compilation.
