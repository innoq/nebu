## Threat Model for This Story

Story 11.3 wires the `Nebu.Search.DB` SQL contract (shipped in 11.2) to a gRPC handler, with the user_id trust-boundary handoff that Story 11.2's Kassandra review explicitly flagged as the moment IDOR becomes exploitable. The headline security goals are:

1. **No IDOR via attacker-controlled `user_id`** — the handler must derive `user_id` from validated gRPC metadata, never from `request.user_id`, even though the proto carries that field.
2. **Authentication enforcement** — unauthenticated callers (no `x-user-id` header) must be rejected with `GRPC.Status.unauthenticated()`, not silently treated as a known user or allowed to query the empty user_id set.
3. **No regression of SQL-layer membership scoping** — the `room_filter` parameter must intersect membership, never replace it.
4. **No DB-error information disclosure** — Postgrex error details (schema names, table names, constraint messages) must not propagate to gRPC clients.
5. **No DOS vectors via unbounded inputs** — limit clamping, offset bounding, search-term sanity, room_filter cardinality.

Dimensions assessed (per CLAUDE.md scope and `references/security-review.md`):

- IDOR / cross-room leakage at the trust-boundary handoff
- Auth bypass on the new RPC entry
- SQL injection in handler-built parameters and the new SQL constant
- DB error leakage to client
- Offset abuse / pagination-token forgery
- Room-filter bypass (replacement vs. intersection)
- Proto-field trust violations (reading `request.user_id`)
- Limit / offset / search_term / room_filter DOS vectors
- Logging hygiene (no secret leak in `Logger.error`)
- Crypto, timing, body-size, rate limits, headers — N/A or out of scope

## Findings

| # | Severity | Location | Vulnerability | Remediation |
|---|----------|----------|---------------|-------------|
| 1 | MEDIUM | `core/apps/event_dispatcher/lib/nebu/event_dispatcher/server.ex:2630-2638` | The `next_batch` pagination token is a base64-encoded plaintext integer offset with **no upper bound and no signature**. After successful decode, any `n >= 0` is accepted as the offset and passed verbatim into the SQL `OFFSET $4` clause. An authenticated attacker can forge `next_batch = Base.encode64("99999999999")` and force PostgreSQL to scan + discard 100 billion matching rows on every request — `OFFSET N` is O(N) even with the GIN index, because PostgreSQL must materialise and skip N rows before producing the first result. With Story 11.5 (rate limiting) not yet shipped, a single authenticated user can drive arbitrary DB load with a tiny request payload — a high-amplification DOS vector. The token also being unsigned means there is no way to detect that the client did not produce it from a previous server response: pagination invariants are decorative, not enforced. | Clamp the decoded offset to a sane upper bound at handler entry: `offset = min(max(decoded_offset, 0), 10_000)`. Matrix server convention caps `from`-tokens at a few thousand for the same reason. As a defence-in-depth, also clamp inside `Nebu.Search.DB.search_messages/5` (per Story 11.2 Kassandra Finding LOW-4) so the DB layer is robust regardless of caller hygiene. The "unsigned token" framing from code-review MINOR-2 is a symptom; the real risk is unbounded-offset DOS, which clamping resolves without needing token signatures. (HMAC-signed tokens would also work but are heavier and not required for the threat model — confidentiality of the offset is not a goal; bounded values are.) |
| 2 | LOW | `core/apps/event_dispatcher/lib/nebu/event_dispatcher/server.ex:2643` | `room_filter = request.room_filter \|\| []` accepts a list of arbitrary cardinality from the request and passes it directly to PostgreSQL via `ANY($5)`. There is no upper bound on the number of room IDs, no length cap on each room ID string, and no validation that they look like Matrix room IDs (`!opaque:server`). gRPC's default max-message-size (~4 MB) caps the absolute worst case, but a 100 000-element array still bloats the SQL plan and stresses the planner. Also, an attacker can probe arbitrary "room_id" strings — the membership subquery still gates results, so this is not an information-disclosure vector, but it is a free-form input being shipped into SQL that nobody validates. | Clamp `room_filter` length at handler entry: `room_filter = Enum.take(request.room_filter \|\| [], 100)`. Optionally validate each entry is a non-empty string starting with `!`. PostgreSQL handles arrays of a few hundred IDs efficiently; thousands begin to stress the planner. Defence-in-depth, not currently exploitable beyond minor planner load. |
| 3 | LOW | `core/apps/event_dispatcher/lib/nebu/event_dispatcher/server.ex:2622-2623` | `search_term = request.search_term` is passed unbounded to `websearch_to_tsquery('pg_catalog.simple', $2)`. `websearch_to_tsquery` is a forgiving parser (correctly chosen — see ADR-010), but it is not free: a 1 MB search term will tokenise 1 MB of input on every query. gRPC's max-message-size and the rest of the request payload provide a soft cap, but no application-layer length check exists. Combined with Finding #1's offset abuse, an attacker can inflate per-request server cost meaningfully. | Add a defensive length cap: `search_term = String.slice(request.search_term \|\| "", 0, 1024)`. Matrix spec does not standardise a `search_term` length; 1 KB is well above any human use case and well below DOS thresholds. |
| 4 | LOW | `core/apps/event_dispatcher/lib/nebu/event_dispatcher/server.ex:2710` | `Logger.error("search_messages failed", user_id: user_id, error: inspect(reason))` logs the full Postgrex reason via `inspect/1`. This is the correct hygiene direction (server-side log, sanitised client response), but `inspect(reason)` on a `Postgrex.Error` will include the SQL parameters in the formatted output — and parameter `$2` is the raw user-supplied search term. If a user searches for, say, an internal ticket number or partial PII string, that string lands in production logs verbatim, indexed forever by whatever log shipper consumes them. Not a vulnerability against the search semantics; a privacy-by-accident issue for log retention. | Log the error code (`reason.postgres.code` for Postgrex errors) and `user_id` only — not the full inspected struct. Or use a structured logger field with explicit allowlist. The current shape is acceptable for MVP debugging; flag for cleanup before production. |
| 5 | LOW | `core/apps/event_dispatcher/lib/nebu/event_dispatcher/server.ex:2616-2620` | The auth check rejects `nil` and `""` user_ids but accepts whitespace-only or non-Matrix-shaped strings (e.g. `" "`, `"alice"`, `"@:"`) without further validation. Since `x-user-id` is set by the Go JWTMiddleware after OIDC validation — Elixir trusts these values fully per `Nebu.Grpc.Metadata` moduledoc — this is theoretical: the gateway is the trust boundary and it produces well-formed Matrix IDs. But if the gateway ever has a bug or a non-OIDC code path leaks a malformed `x-user-id`, the search query will run with that value and either return zero results (best case — empty membership match) or, depending on database state, leak data scoped to whatever user_id was injected. | Defence-in-depth: validate `user_id` matches the Matrix user-ID shape (`^@[^:]+:.+$`) before calling the DB. Or rely on gateway invariant and document the assumption in a moduledoc comment. Not currently exploitable; the pattern would also strengthen every other handler in `server.ex` (which all have the same pattern). |

### Detail

**Finding #1 — Unbounded offset enables high-amplification DOS (MEDIUM)**

```elixir
offset =
  case Base.decode64(request.next_batch || "") do
    {:ok, bin} ->
      case Integer.parse(bin) do
        {n, ""} when n >= 0 -> n   # ← any non-negative integer accepted
        _ -> 0
      end
    :error -> 0
  end
```

Code-review MINOR-2 framed this as "unsigned tokens" — the question of whether a client can produce a `next_batch` value the server did not issue. That is true but secondary. The real issue is the SQL impact:

```sql
ORDER BY rank DESC, e.origin_server_ts DESC
LIMIT $3
OFFSET $4
```

PostgreSQL's `OFFSET N` is O(N): the planner must produce, sort, and discard the first N matching rows. With the GIN index on `search_vector`, the inner scan is index-driven, but the discard cost is linear. Empirical numbers from PostgreSQL docs and pgbench: at `OFFSET 1_000_000` on a moderately sized table, query latency reaches single-digit seconds. At `OFFSET 100_000_000`, tens of seconds. The handler accepts arbitrary integers up to Elixir's `Integer.parse/1` limit (effectively unbounded — Elixir integers are arbitrary precision; the bind into PostgreSQL is `bigint`, capping at ~9.2e18 but everything below that is accepted by Postgres before producing the discard cost).

A small request body — encoded `next_batch` of a dozen bytes — drives an arbitrary database load. With Story 11.5 (rate limiting) not yet shipped, every authenticated user has this primitive. The code-review labelled this MINOR; I am promoting it to MEDIUM because the threat model is "any authenticated user can DOS the database with a single request" and the fix is a one-line clamp.

The token-signing question (HMAC over the offset) is orthogonal: signing prevents the *first* request from using a server-not-issued offset, but a legitimate flow has the server issuing offset=10, offset=20, offset=30 ... up to whatever depth the data supports. If the data has a million matches, the server itself will issue offset=1_000_000 — and the same DOS landing zone exists for the legitimate client. Bounded offsets are the actual control. (A client wanting to walk past offset=10_000 should switch to a `since`-token / cursor-based pagination scheme, which is what Matrix `/messages` does — but Story 11.5 is the right place for that work.)

**Finding #2 — Unbounded `room_filter` cardinality (LOW)**

`room_filter = request.room_filter || []` then passed to SQL `AND e.room_id = ANY($5)`. PostgreSQL handles small arrays cleanly; very large arrays inflate the SQL plan size and the bind-parameter payload but do not change query correctness. The membership subquery still runs (intersection is preserved), so this is not an information-disclosure vector.

**Finding #3 — Unbounded `search_term` length (LOW)**

`websearch_to_tsquery('pg_catalog.simple', $2)` is the right primitive — forgiving, no DOS via parser exceptions — but tokenisation cost is linear in input length. No application-layer cap.

**Finding #4 — Search term may land in logs via `inspect(reason)` (LOW)**

When the DB call fails with a `Postgrex.Error`, `inspect(reason)` will format the struct and may include the bound SQL parameters depending on the error variant. Parameter `$2` is the raw search term. This is privacy-by-accident: search terms can contain partial PII (names, ticket numbers, internal IDs). Logs are typically retained longer than the search index would surface them. Not a security boundary breach; a log-hygiene gap to fix before production rollout.

**Finding #5 — Auth check accepts malformed-but-non-empty user_id (LOW)**

The `is_nil/empty` check is correct against the documented contract. Anything stricter is defence-in-depth for the case where the gateway invariant breaks. The pattern matches the rest of `server.ex`, so this is a project-wide hardening question, not a 11.3-specific defect.

## Areas Reviewed and Found CLEAN

- **IDOR via `request.user_id` (the headline threat):** Verified by inspection — handler line 2614 reads `{user_id, _system_role} = Nebu.Grpc.Metadata.trusted_identity(stream)` and never references `request.user_id`. Test 1 (`AC2 — user_id comes from metadata, not request field (security regression)`) at `search_messages_grpc_test.exs:109-126` is the regression sentinel: it sets `request.user_id = "@mallory:test.local"` while metadata contains `@alice:test.local` and asserts the FakeSearchDB was called with the metadata user. The test is the explicit "this must never silently break" anchor demanded by Story 11.2 Kassandra Finding MEDIUM-2. ✓

- **Auth bypass on unauthenticated callers (AC4):** `is_nil(user_id) or user_id == ""` raises `GRPC.RPCError` with `GRPC.Status.unauthenticated()` before any DB call. Test 2 (`AC4 — unauthenticated request returns GRPC.RPCError UNAUTHENTICATED`) verifies. The handler cannot reach `search_db_module().search_messages/5` without a non-empty `x-user-id`. ✓

- **SQL injection in the new SQL constant (`@sql_search_messages_with_room_filter`):** All five parameters (`user_id`, `term`, `limit`, `offset`, `room_filter`) are bound through `Ecto.Adapters.SQL.query/3` — zero string interpolation. The SQL constant is a static module attribute. The new `AND e.room_id = ANY($5)` clause uses the standard Postgres array-bind primitive, which is parameterised end-to-end. ✓

- **SQL injection in handler-built fields:** No SQL is built in `search_messages/2`. All values flow through `Nebu.Search.DB.search_messages/5` as bind parameters. ✓

- **Room-filter bypass / replacement (AC5):** Verified by reading `@sql_search_messages_with_room_filter` (`db.ex:41-68`): the membership subquery `e.room_id IN (SELECT room_id FROM room_members WHERE user_id = $1 AND left_at IS NULL)` is preserved, with `AND e.room_id = ANY($5)` added as an *additional* AND-clause. The two filters intersect — they cannot replace one another because both AND-clauses must succeed for a row to match. The `room_filter = []` branch routes to the original SQL constant entirely, so empty filter behaves identically to before. ✓

- **Proto-field trust:** `request.user_id` is mentioned only in the proto-level documentation (`SearchMessagesRequest` field `user_id = 1`), explicitly marked "IGNORED by handler — use x-user-id metadata". Static grep confirmed: `request.user_id` does not appear in the handler body. ✓

- **DB error sanitisation (AC for Test 6):** Error path raises `GRPC.RPCError{status: GRPC.Status.internal(), message: "search failed"}` — generic, no schema detail. Test 6 (`DB error is sanitized — raw error not propagated to client (Kassandra LOW-3)`) seeds a `Postgrex.Error` with `"schema detail leak"` and `"relation events does not exist"`, then `refute String.contains?(err.message, "schema detail leak")` and `refute String.contains?(err.message, "relation")`. This closes Story 11.2 Kassandra Finding LOW-3 at the layer where it matters. ✓

- **Limit clamping (Story 11.2 Kassandra Finding LOW-4):** `raw_limit = if request.limit == 0, do: 10, else: request.limit` then `limit = min(max(raw_limit, 1), 100)`. Negative request.limit becomes 1, zero becomes 10 (default), values >100 are capped. ✓ (The corresponding *offset* clamp is missing — see Finding #1.)

- **Special-character handling in search term (AC3):** `websearch_to_tsquery('pg_catalog.simple', $2)` per ADR-010 — forgiving parser, does not raise on unbalanced quotes, special characters, or unknown operators. Test 5 verifies with `"hello' & ): world:"`. ✓

- **Pagination token base64 decode safety:** `Base.decode64/1` returns `:error` on malformed input — handled. `Integer.parse/1` requires the *entire* string to be numeric for `{n, ""}` to match — a token like `"abc123"` decoded then parsed would yield `{:error, _}` or non-empty rest, falling through to `offset = 0`. Negative integers are rejected by the `n >= 0` guard. ✓ (Subject to Finding #1: positive integers are unbounded.)

- **Test injection mechanism (`search_db_module/0`):** Follows the established pattern of 9 other injectable modules in `server.ex`. `Application.get_env/3` with a sane default of the real `Nebu.Search.DB`. Test isolation correct via `Application.put_env` + `on_exit` cleanup. ✓

- **Crypto primitives:** N/A — no crypto introduced. Base64 here is encoding, not encryption.

- **Timing attacks on secret comparison:** N/A — no secret comparison in this code path.

- **Hardcoded secrets:** None.

- **New SQL migrations:** None in this story (000042 already shipped in 11.1, used by 11.2 and 11.3 unchanged).

- **New tables / RLS:** None.

- **CSRF / open redirects / security headers / XSS:** N/A — no HTTP surface introduced. The Go gateway handler is Story 11.4.

- **Body-size / rate limits:** Out of scope; deferred to Story 11.5 per epic plan. Findings #1–#3 are partially mitigated by Story 11.5's eventual arrival but should not depend on it — the clamps are 3 lines and the DOS vector is exploitable today.

- **Test code (Elixir):** `search_messages_grpc_test.exs` uses `async: false` (correct for `Application.put_env` global state), creates and destroys an ETS table per test, uses parameterised data only. No cookie forging, no DB seeding shortcuts (it operates entirely in-memory via FakeSearchDB). The test for `Postgrex.Error` constructs a synthetic error map matching the real struct shape — no external DB needed. ✓

- **Test code (Go):** The six gateway test files add 5-line `SearchMessages` stubs to satisfy the `pb.CoreServiceClient` interface that the regenerated `core_grpc.pb.go` extends. All return empty responses, no logic. These exist purely to keep existing test mocks compilable after the proto extension. ✓

- **Generated proto stubs (`core.pb.ex`, `core_grpc.pb.ex`, `core.pb.go`, `core_grpc.pb.go`):** Regenerated by `make proto` from `proto/core.proto`. Diffs are mechanical and consistent with `protoc-gen-go` / `protoc-gen-elixir` output for the new message types. No manually injected code in these files (the manual `core_grpc.pb.ex` entry for `SearchMessages` follows the same pattern as the manually backfilled `UpgradeRoom` and `ListAdminRoomMembers` entries — this is documented as a known limitation of `protoc-gen-elixir` in the story completion notes, not a security shortcut). ✓

**Verdict:** APPROVED with one MEDIUM hardening request — no CRITICAL or HIGH findings; commit not blocked.

The two highest-stakes properties of this story — **(a) IDOR prevention at the user_id trust boundary** and **(b) preservation of SQL-layer membership scoping when room_filter is supplied** — are implemented correctly and verified by ATDD tests written before implementation. The headline Kassandra Finding from Story 11.2 (MEDIUM-2: docstring warning that `user_id` must come from session) is now enforced not just by docstring but by a regression test that fails loudly if the handler ever reads `request.user_id` (Test 1).

Findings #2–#5 are defence-in-depth recommendations; none are currently exploitable to a meaningful degree. Finding #1 is genuinely exploitable today by any authenticated user, but the impact ceiling is database load (no data leak, no auth bypass) and the fix is a single-line clamp that should land before Story 11.4 exposes this RPC over HTTP. Story 11.5's rate limiting will reduce but not eliminate the residual risk; bounded offsets are the right control regardless.

## Recommendations Carried Forward

1. **In this story (Finding #1):** Add `offset = min(max(decoded_offset, 0), 10_000)` after the base64 decode block. One line, no behavioural change for legitimate pagination, eliminates the DOS vector. Optionally add the same clamp to `Nebu.Search.DB.search_messages/5` per Story 11.2 Kassandra Finding LOW-4 — defence-in-depth, robust against future callers.

2. **Story 11.4 acceptance criterion add-on:** "The Go gateway HTTP handler for `/_matrix/client/v3/search` MUST cap `room_id_set` cardinality and search-term length before delegating to gRPC." Mirror the clamps from Findings #2 and #3 at the HTTP boundary so they apply even if the gRPC layer is ever called from a non-gateway client.

3. **Story 11.5 acceptance criterion add-on:** Beyond rate limiting, add a per-user offset ceiling enforced at the gateway level (e.g., total OFFSET across pagination calls in a 1-minute window < 100_000). This is the proper long-term control; Finding #1's clamp is the bridge until 11.5 ships.

4. **Cross-cutting log-hygiene cleanup (Finding #4):** `Logger.error` patterns across `server.ex` that pass `inspect(reason)` for Postgrex errors should be audited for log-content disclosure. Not blocking this story; flag for a "log-hygiene sweep" follow-up.

5. **Cross-cutting trust-boundary hardening (Finding #5):** Consider adding a `Nebu.Grpc.Metadata.trusted_identity!/1` variant that validates the user_id matches the Matrix shape and raises `GRPC.Status.unauthenticated()` otherwise. This would harden every handler in `server.ex` against future gateway invariant breakage. Not in scope for 11.3.

## Memory Updates

This review surfaces three patterns worth recording in `_bmad/memory/nebu-agent-kassandra/MEMORY.md`:

- **Unsigned pagination tokens are a DOS vector, not (primarily) an integrity vector.** When reviewing pagination handlers, the question is not "can a client forge a token" — that is usually low-impact — but "can a forged or escalated token drive expensive `OFFSET N` queries". The fix is bounded offsets, not signed tokens. Pattern to scan for: any `OFFSET $N` clause where `N` arrives from client input without a clamp.

- **`inspect(reason)` in `Logger.error` for Postgrex errors leaks bind parameters.** Postgres errors include the SQL parameters in their formatted output. If any of those parameters carry user input (search terms, content fragments, IDs), they land in production logs verbatim. Pattern to scan for: `inspect(reason)` or `inspect(error)` in error-handling logging where the underlying error is a `Postgrex.Error`.

- **Trust-boundary handoff between stories is the highest-risk moment.** Story 11.2's Kassandra Finding MEDIUM-2 ("docstring the user_id contract") was not directly exploitable until Story 11.3 wired the handler. The handoff was the moment IDOR became real. The mitigation worked: the docstring became an AC, became a security-regression test (Test 1), became a structural property the codebase will defend going forward. Pattern to apply: every "this becomes exploitable in the next story" finding should escalate to an ATDD test in that story, not just a docstring update.
