## Key Findings

### Bundled Aggregations — The Root Cause
The spec says bundled aggregations SHOULD be included in `/sync` ONLY when `limited: true`. 
For non-limited syncs (normal incremental sync), bundled aggregations are NOT required by the spec.
BUT: Element Web relies on bundled aggregations in the thread root event to know there IS a thread.
If the thread root arrives in sync without `unsigned.m.relations.m.thread`, Element Web does not
know to show the thread panel — even if the reply event is also in the same sync response.

When the FIRST reply is sent:
- The thread root was already received in a prior sync (no bundled aggregation then, thread didn't exist yet)
- The reply arrives in the next sync — Element Web needs to RE-RECEIVE the thread root with updated
  bundled aggregation to know a thread now exists
- The spec DOES say: "The aggregation provided by the server only includes child events which were
  known at the time the client would receive the aggregation. For example, in a single /sync response
  with the parent and multiple child events the child events would have already been included on
  the parent's m.relations field."
- This means: when the first reply and its thread root both appear in the SAME sync window, 
  the thread root's bundled aggregation MUST include that reply. But if the thread root was sent
  in a PRIOR sync, there's no re-delivery of the thread root — the client only gets the reply event.

### Relations Endpoint — Required for Thread Display
`GET /_matrix/client/v1/rooms/{roomId}/relations/{eventId}/m.thread`
Element Web uses this to load thread timeline. Without it, the thread panel cannot populate.
The spec defines this endpoint (added v1.3, threading added v1.4).

### No Special "First Reply" Handling
The spec does NOT distinguish between first and subsequent replies. The mechanism is identical.
The bug is structural: bundled aggregations on the thread root are not being sent by Nebu.
