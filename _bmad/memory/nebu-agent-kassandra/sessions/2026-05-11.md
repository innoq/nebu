### Checks performed (all CLEAN)

1. user_id context spoofing — `contextKey` is unexported type; context.Value cannot be set from HTTP/gRPC metadata; only JWTMiddleware writes ContextKeyUserID and only after Verify() succeeds.
2. DoS via fallback IP path — unauthenticated requests rejected by `jwtWithStatusCheck` before reaching `searchRL`; fallback unreachable in production. LRU capped at 10 000.
3. Bucket exhaustion via user_id manipulation — same root cause as #1.
4. LRU Get+Add race — `mu.Lock()/defer mu.Unlock()` wraps the compound op (line 224–233); identical to the hardened Story 5.21 pattern.
5. Retry-After info leak — `retry_after_ms` derived purely from bucket arithmetic (delay/rate/burst). Bounded 1–60s. No DB/system timing leak.
6. NEBU_RATE_LIMIT_DISABLED — checked at construction (line 217), not in hot path. No TOCTOU.
7. Chain order — `bodyLimit1MiB(jwtWithStatusCheck(searchRL(handler)))` (main.go:737). JWT runs first → ContextKeyUserID populated → searchRL reads it. Correct.

### Notable design observations

- Story re-used the Story 5.21 `NewIPRateLimiter` pattern faithfully, deviating only where the spec required (key on user_id, emit `retry_after_ms` in body).
- IP fallback is dead code in production but a sound defense-in-depth choice — prevents a panic if a future story accidentally wires the middleware to an unauthenticated endpoint.
- Comment in `ratelimit.go:210-214` explicitly documents the required chain order, which matches the actual wiring in main.go.
