## Session — Story 13-8: Managed Database Services (Aurora Serverless v2 + Stackit PostgresFlex)

**Scope:** Staged diff for story 13-8. AWS module: `aws_db_instance` → `aws_rds_cluster` + `aws_rds_cluster_instance` (Aurora Serverless v2). Stackit example: self-hosted `postgres:16-alpine` in docker-compose → `stackit_postgresflex_instance` with separate `nebu_app`/`keycloak_app` users and ACL-restricted to private network CIDR. Cloud-init template, RUNBOOK, outputs, variables all updated.

**Finding summary:** 0 CRITICAL, 0 HIGH, 2 MEDIUM, 4 LOW

- **MED-1:** URL injection via unencoded auto-generated `pg_password` in `NEBU_DB_URL=postgresql://${pg_user}:${pg_password}@...`. The MINOR-5 deferral comment is documentation-only; provider-generated credentials are not under operator control. Remediation: switch to `PG*` env vars or `urlencode()` in template, or add a `precondition` rejecting reserved chars.
- **MED-2:** `deletion_protection = false` hardcoded on the Aurora cluster. Production-hostile default; should be a variable defaulting to `true`.
- **LOW-3:** `db_password` default `"changeme"` + 8-char minimum. Recommend forcing operator input + 16-char minimum, or switch to `manage_master_user_password = true`.
- **LOW-4:** Missing `iam_database_authentication_enabled = true` on Aurora (defense-in-depth).
- **LOW-5:** `storage_encrypted = true` uses AWS-managed `aws/rds` key, not CMK. Compliance-relevant for PCI/SOC2.
- **LOW-6:** `var.network_cidr` lacks validation rejecting `0.0.0.0/0`; flows directly into PostgresFlex ACL.

**Verdict:** APPROVED (no CRITICAL/HIGH).

**Positive findings worth capturing:**

- Credential isolation improved: shared `db_password` (init-db.sql) → two auto-generated PostgresFlex users (`nebu_app`, `keycloak_app`). Real privilege-separation win.
- `init-db.sql` removed entirely — one fewer secret-bearing artifact on the VM.
- `docker-compose.yml` permissions tightened `0640` → `0600`.
- `KC_DB_PASSWORD` moved from inline compose to `.env` (already `0600`).
- Aurora cluster is inherently VPC-only (no `publicly_accessible` attribute on `aws_rds_cluster`); existing `aws_security_group.rds` only allows 5432 from ECS SG.

**New patterns discovered:**

1. **DSN composition with externally-generated passwords** — recurring infra-layer risk. Any time a provider-generated credential (Stackit PostgresFlex, AWS Secrets Manager auto-rotation, RandomPassword + provider escape semantics) ends up inside a URI (PostgreSQL DSN, AMQP URL, MongoDB connection string), require URL-encoding or switch to env-var form. Add to MEMORY.md as a recurring pattern.

2. **`deletion_protection` defaults** — every database-bearing resource we provision should default to `deletion_protection = true` with explicit dev overrides, not the inverse. Review existing IaC for the same anti-pattern (RDS, Aurora, DynamoDB, EFS, EBS snapshots).

3. **Singleton-list-passthrough variables for SG/ACL inputs** — `acl = [var.network_cidr]` is a footgun if `var.network_cidr` is not validated. Enforce validation rejecting `0.0.0.0/0` and over-broad prefixes wherever a variable flows directly into an allowlist.

**Accepted risks this session:** None — all findings tracked for follow-up.

- The LOW items are appropriate for a Phase-3 production-hardening story.
- Update MEMORY.md with the three new patterns above after this review is acknowledged.
