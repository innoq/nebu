## Recurring Patterns

| Pattern | Epics | Description |
|---------|-------|-------------|
| Missing RLS on new tables | 9 | New tables (forgotten_rooms) added without RLS policy, breaking defense-in-depth. Check every new table migration for ENABLE ROW LEVEL SECURITY + policy. |
| Device-ID threading gaps | 9 | When per-device columns are added to existing queries, all dependent query helpers must be updated to pass device_id. Check all query helpers when schema adds device_id to PK. |
| Nullable state_key + equality filter | 11 | Any `WHERE state_key = '...'` filter misses NULL rows because in three-valued SQL logic `NULL = ''` is NULL. The events.state_key column (mig 000038) is nullable. Defense-in-depth: prefer event_type-only checks for "is room encrypted/redacted/etc.", or include `OR state_key IS NULL` explicitly. |
| DB-module user_id trust-boundary docstring | 11 | New DB modules taking `user_id` for authorization scoping must document loudly that it MUST come from the validated session, not from request payload. The hand-off to gRPC handler stories is the natural spot to lose that invariant — see Story 11.3. |
| Permissive RLS UPDATE without key-scope | 11, 12 | `CREATE POLICY ... FOR UPDATE USING (true) WITH CHECK (true)` on shared key-value tables (e.g., `server_config`) destroys the per-key immutability invariant. Always scope UPDATE policies by `key IN (...)` allowlist. Migration 000045 is the cautionary tale. |
| "// for MVP" auth shortcut → live vulnerability | 4 → 12 | A `// for MVP, accept any bearer` comment lived in media/upload for 8 epics; Story 12.3 wired the service into compose and exposed port 8009. The dormant defect became reachable. Rule: when promoting a code-only module to a deployed compose service, re-audit every `// MVP` / `// placeholder` / `// TODO` token in the affected files. |
| MinIO mc entrypoint secret leak via argv | 12 | `$(cat /run/secrets/x)` in shell entrypoints expands secrets into `mc` argv → recoverable via `/proc/<pid>/cmdline`. Defeats Docker Secrets. Use `MC_HOST_<alias>` env var or `--stdin` form. |
| Unauthenticated media endpoint + unbounded resource | 12 | Thumbnail (`width`/`height`) and download (`io.ReadAll`) handlers do expensive per-request work without auth and without bounds. Every unauthenticated Matrix v3 endpoint MUST have hard caps on input dimensions, output size, and concurrency. |
| Inline Content-Type echoed back with `inline` disposition | 12 (4) | Stored XSS surface: attacker uploads `Content-Type: text/html`, download serves it back inline. Matrix spec v1.12+ requires `attachment` for unsafe types. Add `X-Content-Type-Options: nosniff` always; allowlist server-controlled types on download. |
| OIDC fail-open at startup | 12 | When an OIDC-required service falls back to "any bearer accepted" if the provider can't be reached at boot, an operator misconfiguration (empty issuer env, Dex offline) silently produces an anonymous-upload surface. Pattern: services that require OIDC must support a `_REQUIRED=true` knob to refuse startup when the verifier is nil. Logged `slog.Warn` is necessary but insufficient. |
| `uploader_user_id` ≠ Matrix user ID | 12 | When media stores identity claims (raw `sub` or `name`) without going through the gateway's `FormatUserIDFromClaims` translation, audit-trail correlation breaks. Any service that records "user IDs" must store the canonical Matrix `@localpart:server` form, not raw OIDC claims. Story 12.9 partially closed this: added `@localpart:server` format + sanitisation, but still extracts from `claims.Sub` (with `Name` fallback) instead of operator-configured `oidc_user_id_claim`. Full fix requires reading that config (DB or env) at media startup. |
| XFF rate-limit spoofing on directly-exposed services | 12 | When a service exposes a port directly to the host (`ports: ["8009:8009"]`) and the rate-limiter trusts `X-Forwarded-For` without a `trustedProxy` toggle, attackers rotate the rightmost XFF entry to force fresh sync.Map buckets and bypass per-IP rate limits entirely. Always gate XFF parsing behind an explicit `NEBU_TRUSTED_PROXY` env (default `false`), mirroring `gateway/internal/middleware/ratelimit.go` (Story 12.10 missed this). |
| Per-story CLEAN reviews can stack into combined HIGH at epic-end | 12 | 12.8, 12.9, 12.10 each reviewed CLEAN in isolation. Together they created (a) audit-correlation defeat and (b) bypassable rate-limit on shipping topology. Per-story reviews must consider deployment topology, not just the diff in isolation. Epic-end review is non-redundant. |
| Deprecated CSP directive `plugin-types` misleads auditors | 12 | `plugin-types application/pdf` is ignored by all modern browsers (removed from CSP3 spec; Firefox 84+, Chrome 70+ do not implement it). Including it in a CSP header creates false assurance that PDF plugin embedding is controlled by a directive when it is actually controlled (correctly) by `object-src`. Remove from all CSP headers; rely on `object-src 'self'` or `object-src 'none'` instead. |
| Unvalidated URL path segment in Content-Disposition filename | 12 | When a URL path variable (e.g. `{fileName}`) is used as a `Content-Disposition` filename, apply the same input-character allowlist used elsewhere in the service (e.g. `mediaIDPattern: ^[A-Za-z0-9_\-]+$`). Go's `%q` prevents CRLF injection but does not produce RFC 5987-safe filenames and does not strip semantically dangerous characters from log/audit perspectives. |
| Story-title oversell on partial graceful-shutdown | 12 | Story 12.13 titled "Graceful Shutdown" but only wires signal-awareness into OIDC startup; `http.Server.Shutdown(ctx)` is NOT called, so SIGTERM during serving phase still results in Docker SIGKILL after 10s grace. Not a security finding (no auth bypass, no data exposure), but a pattern to watch: future "graceful shutdown" claims must verify `srv.Shutdown(ctx)` with a `shutdownTimeout` is actually wired, plus a regression test that issues a request → SIGTERM → asserts request completes before exit. **CLOSED 2026-05-13 by Story 12.14**: `srv.Shutdown(shutdownCtx, 30s)` + `pool.Close()` + ctx-aware `cleanupLoop` all wired; pattern remains as a watch-flag for future "graceful X" stories. |
| Shutdown-phase auth bypass via cancelled request ctx | (audit) | In Go HTTP servers, `r.Context()` derives from `srv.BaseContext` (default `context.Background()`), NOT from any outer SIGTERM ctx. This is actually a feature: in-flight requests during `srv.Shutdown` complete their full middleware stack with uncancelled request contexts — auth/rate-limit/OIDC verification still run. BUT: if a future story sets `srv.BaseContext = func(_ net.Listener) context.Context { return shutdownCtx }` to "speed up drain", it would propagate cancellation into request contexts and could cause `oidcVerifier.Verify(r.Context(), ...)` to error with `context.Canceled` → 401 → and depending on error handling, potentially leak a bypass. Rule: never wire the outer SIGTERM ctx as `BaseContext` without also auditing every `ctx.Err()` check in the request path. |
| DSN composition with externally-generated passwords | 13 | Provider-generated credentials (Stackit PostgresFlex, AWS Secrets Manager auto-rotation, etc.) inserted directly into URIs (postgres://, amqp://, mongodb://) without URL-encoding can produce parser-level auth bypass when reserved chars (`@:/?#[]%`) appear. Fix: switch to env-var form (PGHOST/PGUSER/...) or use `urlencode()` in templates. Documentation comments are NOT a substitute. |
| `deletion_protection = false` hardcoded defaults | 13 | Database-bearing IaC resources (RDS, Aurora, DynamoDB, EFS, EBS snapshots) frequently ship with `deletion_protection = false` baked into module code. Production must default to `true` with an explicit dev override variable, not the inverse. Check every db/store resource on touch. |
| Singleton-list passthrough into ACL/SG inputs | 13 | Patterns like `acl = [var.network_cidr]` are footguns: if the source variable lacks a `0.0.0.0/0` and overly-broad-prefix validation, a single tfvars copy-paste accident exposes the resource. Validate at the variable, not at the resource. |
| Admin-configured outbound URL = SSRF + credential surface | 14 (pre-impl) | Any feature where admin stores an external endpoint URL (OIDC directory, SCIM, webhooks, notifications) creates two surfaces: (1) SSRF if private IPs not blocked, (2) credential exposure if the auth token for that endpoint appears in logs/API responses. Both surfaces require explicit mitigation on every new feature of this type. |
| Long-running op progress endpoint needs auth gate | 14 (pre-impl) | A progress/status endpoint for a long-running admin operation (bulk import, migration, export) leaks operational state (org size, import counts). Must be behind admin auth middleware — not just the trigger endpoint. Verify router chain, not handler. |
| ContextKeySub vs ContextKeyUserID identity confusion | 11, 12, 14 | Handlers comparing "is this caller equal to this target user?" must compare like-shaped identifiers. `ContextKeySub` is the raw OIDC `sub` (e.g. `kai`, UUID). `ContextKeyUserID` is the canonical Matrix `@localpart:server`. They will never match for the same human. Self-delete / self-anonymize / self-role-change guards must use `ContextKeyUserID` when the path/body parameter is a Matrix user_id. Epic 14 Story 14.4 repeated the pattern first introduced in Story FB-58-03 (`user_anonymization.go`) and broke the four-eyes invariant for `DELETE /api/v1/admin/users/{userId}`. Defense: typed wrapper structs (`MatrixUserID` vs `OIDCSub`) so `go vet` can catch the comparison. |
| gRPC handler ignoring `stream` metadata = anonymous-RPC bypass | 14 | Every public Core gRPC handler whose method signature is `def name(req, stream)` MUST extract `Nebu.Grpc.Metadata.trusted_identity(stream)` and reject the call if the identity is missing/untrusted. `update_server_config/2` does it correctly; `bulk_import_users/2` (Story 14.3a) signs the call with `_stream`, accepting any caller anonymously. Combined with caller-controlled `system_role` in the proto, this becomes a one-RPC admin-account-creation primitive on any deployment where Core gRPC port is reachable beyond the PSK/mTLS boundary. Lint rule: flag every `def …(_, _stream)` or `def …(_, _)` in `server.ex`. |
| Caller-controlled `system_role` in admin gRPC request payload | 14 | When a bulk/admin RPC accepts `system_role` as a free-form string in the proto payload, the value must be either hard-coded server-side (e.g. always `"user"`) or validated against an allowlist after extracting the trusted caller identity. Story 14.3a's `OIDCUserClaims.system_role` is unfiltered all the way through to `users.system_role`, whose CHECK constraint accepts `instance_admin`/`compliance_officer`. Caller picks their own role. |
| Bulk operation with no audit_log entry | 14 | Single-user admin operations (deactivate, anonymize, role-change, delete-keys) all emit audit_log rows. The first bulk operation in the codebase (`Nebu.Session.BulkImporter.import_users/1`, Story 14.3a) does not. Audit-completeness invariants must be carried into new "shape" of action, not just new "type" of action. Add the action name to `@known_actions` in `audit_writer.ex` at the same time the handler is written. |
| Lazy validation of admin-saved external URLs widens SSRF window | 14 | When an admin-saved URL is HTTPS-validated only at call-time (`FetchUsers`) and not at save-time (PATCH /admin/config), the misconfiguration is silent until the feature is exercised. `scim_base_url` is validated at both layers; `oidc_directory_endpoint` only at call-time. Validate at the persistence boundary (API/form handler), not only at the consumer. |
| Asymmetric secret-handling across sibling features | 14 | When two features expose "store a bearer token to call an external endpoint" (Story 14.2b OIDC directory vs Story 14.3c SCIM), the encryption-at-rest, write-only UI shape, and RLS allowlist must match. Story 14.3c does it correctly (AES-256-GCM, write-only password input, `[SET]` placeholder, RLS row). Story 14.2b's `oidc_directory_bearer_token` is read at startup but never writable through UI/API/migration AND has no encryption path — if eventually wired by an operator with direct DB access, it lands plaintext. Pick one pattern and use it for every token of this shape. |
| `/admin/*` security-headers carve-out misses `/api/v1/admin/*` | 4 → 14 | `SecurityHeadersMiddleware` is wrapped only on paths starting with `/admin`. New endpoints under `/api/v1/admin/*` (Story 14.4 GDPR delete, 14.3c import-status) inherit no `X-Content-Type-Options`, `X-Frame-Options`, CSP, or `Referrer-Policy`. Modern browsers don't sniff JSON Content-Type, so the practical XSS risk is low, but the invariant is "every admin response carries these headers". Fix at the routing level (`HasPrefix("/admin") \|\| HasPrefix("/api/v1/admin")`). |
